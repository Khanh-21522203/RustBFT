# DevOps and Deployment

## Purpose

Provides Docker-based multi-node local cluster setup, Prometheus/Grafana monitoring stack, and operational scripts for testing cluster behavior including crash recovery, network partitions, and latency injection.

## Scope

**In scope:**
- `devops/Dockerfile` — multi-stage production image build
- `devops/docker-compose.yml` — 4-node cluster with Prometheus and Grafana
- `devops/scripts/init-cluster.sh` — cluster initialization helper
- `devops/scripts/submit-tx.sh` — JSON-RPC tx submission helper
- `devops/scripts/simulate-failure.sh` — crash/partition/latency simulation
- `devops/prometheus.yml` — Prometheus scrape configuration

**Out of scope:**
- Kubernetes/Helm deployment (not present)
- CI/CD pipeline configuration (not present)
- TLS/cert management (not present)
- Production key management (node keys auto-generated on first start)

## Primary User Flow

**Start 4-node local testnet:**
```
cd devops
./scripts/init-cluster.sh
```
This runs `docker-compose up -d` on 4 `rustbft-node` containers + Prometheus + Grafana.

**Check cluster status:**
```
./scripts/simulate-failure.sh status
```
Polls `/health` on all four RPC ports (26657, 26756, 26856, 26956).

**Submit a transaction:**
```
./scripts/submit-tx.sh http://localhost:26657 <hex-encoded-tx>
```
POSTs a `broadcast_tx` JSON-RPC call to the given node.

## System Flow

```
devops/Dockerfile (multi-stage)
    Stage 1 (rust:1.75-bookworm):
        cargo build --release --bin RustBFT
    Stage 2 (debian:bookworm-slim):
        apt: ca-certificates libclang-dev
        COPY target/release/RustBFT → /usr/local/bin/rustbft-node
        EXPOSE 26656 26657 26660
        ENTRYPOINT ["rustbft-node"]
        CMD ["/config/node.toml"]

docker-compose.yml (devops/docker-compose.yml)
    Network: rustbft-net (bridge, 172.28.0.0/16)

    node0  172.28.1.1  ports: 26656,26657,26660 → host
    node1  172.28.1.2  ports: 26756(rpc),26760(metrics) → host
    node2  172.28.1.3  ports: 26856(rpc),26860(metrics) → host
    node3  172.28.1.4  ports: 26956(rpc),26960(metrics) → host

    Each node:
        volume ./config/nodeN:/config  (node-specific TOML + key)
        volume nodeN-data:/data        (RocksDB + WAL, named volume)
        RUSTBFT_NODE_ID=nodeN
        RUST_LOG=... (node0 uses debug for consensus)
        restart: unless-stopped

    prometheus  172.28.1.10  port 9090
        config: ./prometheus.yml → scrapes 172.28.1.1-4:26660 every 5s
    grafana     172.28.1.11  port 3000
        provisioning: ./grafana/provisioning
        dashboards: ./grafana/dashboards
        GF_AUTH_ANONYMOUS_ENABLED=true (no login needed)
```

## Data Model

**Node config volume layout** (`devops/config/nodeN/`):
- `node.toml` — per-node `NodeConfig` TOML (seeds point to other containers by hostname)
- `node_key.json` — ed25519 keypair; if missing, auto-generated by node on first start

**Named Docker volumes** (survive `docker-compose down`, destroyed on `down -v`):
- `nodeN-data` — maps to `/data` inside container: `blocks/`, `state/`, `consensus.wal`, `node_key.json`
- `prometheus-data`, `grafana-data` — persistent metrics state

**Port mapping:**

| Node  | P2P (internal) | RPC (host)  | Metrics (host) |
|-------|----------------|-------------|----------------|
| node0 | 26656          | 26657       | 26660          |
| node1 | 26656          | 26756       | 26760          |
| node2 | 26656          | 26856       | 26860          |
| node3 | 26656          | 26956       | 26960          |

All P2P ports are only exposed on node0 from the host; nodes communicate via `rustbft-net` bridge.

## Interfaces and Contracts

**`init-cluster.sh`:**
- Checks for existing `devops/config/nodeN/node_key.json`; if absent, notes it will be auto-generated on first start
- Calls `docker-compose up -d`
- Sleeps 5 seconds then polls `/health` on all four RPC ports

**`submit-tx.sh <url> <hex_tx>`:**
- `$1` — node RPC URL (default `http://localhost:26657`)
- `$2` — raw hex-encoded transaction bytes (default `0102030405`)
- POSTs JSON-RPC `broadcast_tx { "tx": "<hex>" }` and pretty-prints response

**`simulate-failure.sh <action> [node]`:**

| Action      | What it does |
|-------------|--------------|
| `crash`     | `docker kill <node>` (default: `rustbft-node2`) |
| `restart`   | `docker start <node>` |
| `partition` | `docker network disconnect` for node2 and node3 |
| `heal`      | `docker network connect` for node2 and node3 |
| `slow`      | `docker exec tc qdisc add … netem delay 200ms` on node's `eth0` |
| `unslow`    | `docker exec tc qdisc del dev eth0 root` |
| `status`    | Polls `/health` on all four RPC ports |

**Prometheus scrape config (`devops/prometheus.yml`):**
- `scrape_interval: 5s` — polls all four nodes at `172.28.1.1-4:26660/metrics`
- Label: `cluster=local-testnet`

## Dependencies

**External tools (not managed by this repo):**
- Docker Engine (for build and container runtime)
- `docker-compose` (v1 or `docker compose` v2)
- `curl` — used in scripts for health checks and tx submission
- `python3 -m json.tool` — used in `submit-tx.sh` for pretty-printing (optional; falls back to `cat`)
- `tc` (iproute2) — inside containers for `slow/unslow`; requires `NET_ADMIN` capability (not explicitly granted in `docker-compose.yml` — `slow` command will fail without it)

## Failure Modes and Edge Cases

- **`slow` action requires `NET_ADMIN` capability:** Not configured in `docker-compose.yml` — `tc qdisc add` will fail with permission denied inside containers.
- **P2P not exposed for node1-3:** Only node0 exposes P2P port `26656` to the host. External peers cannot connect to node1-3 directly.
- **Node key auto-generation on first start:** Each node generates its own keypair independently; no cluster-wide key ceremony. Node IDs are not pre-shared — seed addresses in `node.toml` must use hostnames (`node0`, `node1`, etc.) not pre-known public keys in the seed format.
- **`docker-compose down` without `-v`:** Named volumes persist — RocksDB state survives restart. Use `docker-compose down -v` to start fresh.
- **`init-cluster.sh` does not create config files:** `devops/config/nodeN/node.toml` must exist before running. If absent, node falls back to `NodeConfig::default()` and uses `data/` relative to working directory.
- **Grafana anonymous access:** `GF_AUTH_ANONYMOUS_ENABLED=true` — dashboards are public on port 3000. `GF_SECURITY_ADMIN_PASSWORD=admin` is the admin password.

## Observability and Debugging

- Grafana at `http://localhost:3000` — dashboards in `devops/grafana/dashboards/`
- Prometheus at `http://localhost:9090` — query `rustbft_consensus_height` per node
- Node0 runs with `RUST_LOG=rustbft_consensus=debug,rustbft_p2p=info`; nodes 1-3 run at `info`
- Container logs: `docker logs rustbft-node0` (or `docker-compose logs node0`)
- Simulate node crash and verify remaining 3 of 4 continue committing blocks (BFT threshold: f=1, need 3 honest)

## Risks and Notes

- The `partition` action disconnects node2 and node3 from the network — neither side has >2/3 of voting power (only 2 of 4 validators each), so consensus halts during partition. After `heal`, commits should resume.
- Node keys are auto-generated per container on first start — if the container is destroyed without `-v`, the key is lost and the node gets a new identity, which breaks the validator set.
- `Dockerfile` pins `rust:1.75-bookworm` — build environment may be outdated relative to current toolchain in `rust-toolchain.toml` (if present).
- No health-check directive in `docker-compose.yml` — Compose does not wait for node readiness before declaring the service healthy.
